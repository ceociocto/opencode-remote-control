# Design: OpenCode Remote Control via Telegram & Feishu

## Problem Statement

OpenCode runs on my local machine, but I want to control it from anywhere. When I'm commuting, at a coffee shop, or away from my desk, I should be able to send prompts, check progress, review changes, and approve actions — all through familiar IM apps like Telegram and Feishu.

## What Makes This Cool

- **Code from your pocket.** Stuck on a train? Start a coding task. Waiting in line? Check on your agent's progress.
- **No new app to learn.** You already live in Telegram/Feishu. OpenCode becomes a contact you message.
- **Approval workflow.** Review diffs and approve/reject edits right in chat. You're always in control.
- **Push notifications.** Get pinged when tasks finish, fail, or need your input.

## Design Decisions (from Eng Review)

1. **SDK Research First:** Study `packages/slack` source before implementation to understand session management API.
2. **Session Persistence:** Defer to research — match Slack package's approach.
3. **Handler Abstraction:** Add `core/handler-common.ts` for shared handler logic (DRY across platforms).
4. **Command Routing:** Use platform-native commands (`bot.command()`) for Phase 1.
5. **Session Scope:** One thread = one session. Multiple threads can have parallel sessions.
6. **Response Streaming:** Batch into ~500 char chunks, send every 1-2 sec for progress visibility.

## Constraints

1. **Local machine.** OpenCode runs on your machine at home/work, not in the cloud.
2. **Cloudflare tunnel.** Use Cloudflare Tunnel (cloudflared) to expose the bot webhook endpoint.
3. **Telegram first, Feishu second.** Phase 1 ships Telegram support. Phase 2 adds Feishu.
4. **Follow Slack pattern.** Model the architecture after OpenCode's existing `packages/slack` integration (confirmed exists at `github.com/anomalyco/opencode/tree/dev/packages/slack`).

## Premises

1. **OpenCode's Slack package provides a working blueprint.** The `@opencode-ai/slack` package creates threaded conversations, manages sessions per thread, and connects via the OpenCode SDK. We can follow this same pattern.
2. **Cloudflare Tunnel solves NAT/firewall issues.** Your local machine doesn't need a public IP — cloudflared creates a secure tunnel for webhook delivery.
3. **Approval workflow is secure enough for personal use.** Your Telegram/Feishu account is the authentication layer. If someone gets access to that, they could already impersonate you.

## Approaches Considered

### Approach A: Parallel Package (Selected)
Model a new `packages/telegram-feishu` package after the existing Slack integration. Follow the same patterns: threaded sessions, OpenCode SDK integration, webhook handling.

**Effort:** M (human: 1-2 days / CC: ~1 hour)
**Risk:** Low

### Approach B: Standalone Bridge Service
Build a separate service that connects to OpenCode's client/server API from outside. No modification to OpenCode itself.

**Effort:** M (human: 2-3 days / CC: ~2 hours)
**Risk:** Medium (reverse-engineering required)

### Approach C: Hybrid Multi-Platform Framework
Fork the Slack package and generalize it into a multi-platform bot framework (Telegram, Feishu, Slack, Discord).

**Effort:** L (human: 3-5 days / CC: ~3 hours)
**Risk:** Medium

## Recommended Approach

**Approach A: Parallel Package**

Follow the Slack package blueprint:
1. Create `packages/telegram-feishu` in the OpenCode monorepo (or as a standalone project)
2. Implement Telegram bot using `grammy` or `telegraf`
3. Implement Feishu bot using Feishu Open Platform SDK
4. Connect to OpenCode SDK for session management
5. Use Cloudflare Tunnel for webhook delivery

## Message Design System

### Message Hierarchy (applies to all bot messages)

```
┌─────────────────────────────────────────────────────────┐
│ 1. STATUS INDICATOR (emoji + 1-3 words)                │
│    What just happened / what's needed                  │
├─────────────────────────────────────────────────────────┤
│ 2. CONTEXT (2-4 lines max)                             │
│    Summary of what, files affected, key details        │
├─────────────────────────────────────────────────────────┤
│ 3. ACTIONS (what can the user do?)                     │
│    Available commands, buttons, or "no action needed"  │
└─────────────────────────────────────────────────────────┘
```

### Emoji Vocabulary

| Category | Emoji | Meaning |
|----------|-------|---------|
| Status | ✅ | Success / done |
| Status | ❌ | Error / failed |
| Status | ⏳ | In progress / loading |
| Status | 🤔 | Needs user input |
| Status | 📝 | Awaiting approval |
| Files | 📄 | File summary |
| Files | 🔧 | Code change |
| System | 🚀 | Bot started |
| System | 💤 | Session expired |

### Message Templates

#### Task Completed
```
✅ Done

📄 {X} files changed:
• {file1} (+{add}, -{del})
• {file2} (+{add})

💬 Reply to continue
/files — see all changes
```

#### Task Failed
```
❌ Error: Cannot find module 'foo'

The task failed after 23s. OpenCode is still running.

💬 Try rephrasing your request, or /reset to start fresh
```

#### Awaiting Approval
```
📝 Approval needed: Edit 2 files

📄 Changes:
• src/auth.ts (+12, -3)
• config.json (+1)

/approve — allow changes
/reject — deny changes
/diff — see full diff first
⏱️ Expires in 5 min (auto-reject)
```

#### Needs Input
```
🤔 Question

I found 3 files named 'config.ts':
1. src/config.ts
2. test/config.ts
3. legacy/config.ts

Which one should I use? Reply with the number.
```

#### Bot Started
```
🚀 OpenCode Remote Control ready

💬 Send me a prompt to start coding
/help — see all commands
/status — check OpenCode connection
```

#### Session Expired
```
💤 Session expired (30 min idle)

💬 Send a new message to start fresh
```

### Formatting Rules

1. **Code blocks** — Use \`\`\` for diffs, file paths, and code snippets
2. **Bold** — Use **text** for file names in summaries (never entire messages)
3. **Line length** — Keep under 60 chars for mobile readability
4. **Lists** — Use `•` bullet, not `-` (cleaner on mobile)
5. **Numbers** — Use `1.` `2.` `3.` when user needs to pick an option

### Mobile-First Design

- All messages designed for 375px viewport (iPhone SE width)
- Code blocks scroll horizontally, never wrap
- Long diffs split into multiple messages (Telegram 4096 char limit)
- No wide tables — use vertical lists instead

### Accessibility

| Concern | Solution |
|---------|----------|
| **Screen readers** | Emoji followed by text: "✅ Done" not just "✅" |
| **Colorblindness** | Never rely on color alone — use emoji + text + structure |
| **Motor impairment** | Commands are short: `/approve`, `/reject`, `/diff` (single tap) |
| **Cognitive load** | One action per message when possible |
| **Vision impairment** | High contrast text, avoid tiny text in code blocks |

**Example accessible message:**
```
✅ Done (success indicator + text)

📄 3 files changed: (icon + text, not color alone)
• src/auth.ts (+12, -3)
• src/utils.ts (+5)

💬 Reply to continue (action is clear text, not just emoji)
```

**Anti-pattern (avoid):**
```
✅ ← screen reader reads "check mark" with no context
3 files changed ← no icon, colorblind users miss nothing
src/auth.ts ← no structure, hard to scan
```

### Emotional Design Principles

1. **No silent failures** — Every user action gets a response within 2 seconds
2. **Progress visibility** — Long operations show streaming, not "..." then boom
3. **Explicit control** — Approvals are explicit, not assumed
4. **Graceful degradation** — Offline/error states are honest and helpful
5. **Context persistence** — Session timeout tells user exactly what happened

## Features

### 1. Chat with OpenCode
- Send a message to the bot → OpenCode receives it as a prompt
- Stream responses back chunk by chunk (or in bursts for IM-friendly delivery)
- Each thread = one OpenCode session (like Slack package)

### 2. Push Notifications
- Task completed: "✅ Your refactor is done. 3 files changed."
- Task failed: "❌ Error: Cannot find module 'foo'"
- Needs input: "🤔 I found 3 files named 'config.ts'. Which one?"

### 3. Review Changes
- Bot sends a summary: "📄 Changed files:\n• src/auth.ts (+12, -3)\n• src/utils.ts (+5)"
- Send `/diff` to see the actual diff in a code block
- Send `/files` to list all modified files

### 4. Approve Actions
- When OpenCode wants to write files: "📝 OpenCode wants to edit 2 files. Approve?"
  - `/approve` — allow the edits
  - `/reject` — deny the edits
  - `/diff` — see what will change first
- Same pattern for shell commands

## Interaction State Table

| Feature | LOADING | EMPTY | ERROR | SUCCESS | PARTIAL |
|---------|---------|-------|-------|---------|---------|
| Send prompt | ⏳ "Thinking..." | — | ❌ Error msg | ✅ Response stream | — |
| Session start | ⏳ "Connecting..." | 💤 "No active session" | ❌ "OpenCode offline" | 🚀 "Ready" | — |
| Approval request | ⏳ "Checking..." | 🤔 "Nothing to approve" | ❌ "Request expired" | ✅/❌ Result | 📝 Multiple queued |
| Diff view | ⏳ "Loading diff..." | 📄 "No changes" | ❌ "Diff unavailable" | 📄 Diff in code block | — |
| File list | ⏳ "Scanning..." | 📄 "No files changed" | ❌ "Scan failed" | 📄 File list | — |

### State Details

#### LOADING States
- **Prompt sent:** `⏳ Thinking...` (with typing indicator via `sendChatAction`)
- **Session connecting:** `⏳ Connecting to OpenCode...`
- **Approval check:** `⏳ Checking pending requests...`

#### EMPTY States
- **No active session:** `💬 Send a message to start a session`
- **Nothing to approve:** `🤷 Nothing pending approval right now`
- **No files changed:** `📄 No files were modified`

#### ERROR States
- **OpenCode offline:** See "OpenCode offline" template below
- **Approval expired:** `⏰ Approval request expired (5 min timeout). Changes rejected.`
- **Rate limited:** `⚠️ Too many requests. Please wait 30 seconds.`
- **Network error:** `❌ Connection lost. Retrying in 10s...`

#### SUCCESS States
- **Task done:** See "Task Completed" template
- **Approved:** `✅ Approved — changes applied`
- **Rejected:** `❌ Rejected — changes discarded`

#### PARTIAL States
- **Multiple approvals queued:**
  ```
  📝 3 approval requests pending

  1. Edit src/auth.ts (waiting)
  2. Run npm install (waiting)
  3. Edit config.json (waiting)

  Current: 1. Edit src/auth.ts
  /approve | /reject | /diff | /next
  ```

## Error Handling

| Scenario | Strategy |
|----------|----------|
| Cloudflare Tunnel down | Bot logs error, retries connection every 30s, sends notification when restored |
| Telegram API rate limit | Queue messages, respect `retry_after`, batch responses |
| OpenCode not running | Return "OpenCode is offline" message to user |
| Session timeout | Sessions expire after 30 min idle, user notified |
| Network interruption during approval | Approval request times out after 5 min, defaults to reject |

## Edge Cases

1. **Concurrent messages in same thread** — Queue them, process sequentially
2. **User edits/deletes message** — Ignore edits, deletions have no effect on processed messages
3. **Message too long** — Split into multiple messages, use "continued..." indicator
4. **Multiple pending approvals** — Queue them, process in order, each with unique ID
5. **User doesn't respond to approval** — 5 min timeout, default reject, notify user

## Session Management Details

- **Storage:** In-memory `Map<threadId, Session>`
- **Idle timeout:** 30 minutes (configurable via `SESSION_IDLE_TIMEOUT_MS`)
- **Cleanup:** Timer runs every 5 minutes, removes expired sessions
- **Max sessions:** No hard limit (cleanup prevents unbounded growth)

```typescript
// Cleanup logic
setInterval(() => {
  const now = Date.now()
  for (const [threadId, session] of sessions.entries()) {
    if (now - session.lastActivity > SESSION_IDLE_TIMEOUT_MS) {
      sessions.delete(threadId)
    }
  }
}, CLEANUP_INTERVAL_MS)
```
